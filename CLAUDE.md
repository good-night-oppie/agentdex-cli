## Why agentdex-cli is the Hermes retrofit

agentdex-cli is the **library boundary** between the agentdex Pokédex pattern
(Three Cards: TaskCard / ResultCard / EvolutionCard) and the Hermes
gateway-per-profile plugin runtime. The retrofit decision is documented in
**ADR-0008 §D** (Hermes gateway as the per-profile process boundary) and
**ADR-0009** (the unifying meta-ADR landed at `docs/adr/0009-kaos-substrate-and-retrofit-framing-pokedex-pivot.md`).

Bridges to subscription baselines (Claude Code, Codex app-server, Manus
Camofox) sit at `packages/adx_bridges/`. The orchestrator that drives them is
the Hermes plugin at `packages/agentdex_plugin/`. The plugin is discoverable
via the `hermes_agent.plugins` entry-points group and — as of phase-9 PR-C —
**wired**: `register(ctx)` forwards 5 tools (`agentdex_run_expedition` +
4 registry/routing tools) to `ctx.register_tool` under `toolset="agentdex"`,
so `hermes chat -t agentdex --yolo` drives the Pokédex flow autonomously —
no custom Runner / SessionRunner abstraction needed
(SessionRunner is documented vapor; see phase-4 STATE.md Notable event).

## Why KAOS lives at `packages/kaos/`, not pip install

The vendored copy at `packages/kaos/` is a **git subtree** (squashed +
culled) rather than a published pypi dependency for two reasons:

1. **ACE-FCA (Agent Context Engineering)** — agentdex-cli wants the full
   KAOS source readable from the agent's working tree at all times.
   `lib/.venv/` indirection breaks that reading discipline; every bridge,
   oracle, and expedition orchestrator needs to be able to grep through the
   KAOS substrate without leaving the workspace.
2. **Velocity** — at MVP we want to land doctrine / instrumentation changes
   in KAOS as part of the same commit that exercises them. A pypi
   dependency forces a publish-step decoupling that's premature for the
   first 5 milestones.

The subtree was assembled at phase-4 (commits `2dc26f1` merge + `5039a12`
squash + `3d5540f` cull, ~150MB demo/doc material trimmed). When KAOS hits a
stable v1.0, we'll consider re-extracting to pypi.

## Why helios stays external

`helios` is the Go daemon that maintains the hot tier of mutation-seed
substrate. agentdex-cli vendors only its Python client at
`packages/helios_client/` because:

- helios has its own build/release cycle (Go modules, separate ABI)
- the daemon is a process you operate, not code you import
- M6 (post-MVP) wires helios as the upstream of `SqliteCheckpointStore`;
  M5 still runs against the SQLite fallback so the MVP is buildable without
  a live helios

## Why `~/gh/agentdex/` was archived

The `~/gh/agentdex/` repo was a phase-2 exploration of a FastAPI + VEX
orchestrator. That exploration informed the engine extract but does not
ship: the engine moved to `packages/agentdex_engine/` here and the
FastAPI/VEX surface was retired. The `~/_attic/agentdex-pre-cli/` archive is
the source of truth for "what was tried"; the live engine is here.

## Two-tier substrate (M5 → M6 plan)

- **Hot tier (M6+):** `helios` Go daemon, in-memory + WAL persistence,
  short-window seed scoring + mutation acceptance. Powers the per-Expedition
  EvolutionCard mutation-seed flow.
- **Warm/cold tier (M5):** KAOS sqlite + blob storage. Stores Expedition
  lineage entries (one KAOS agent per Expedition, EvolutionCard payload as a
  set_state JSON blob, snapshot via `checkpoint(label="expedition-final")`).
  Surfaced via `agentdex_engine.shared.kaos_adapter.log_expedition_lineage`.

The two tiers share the `CheckpointStore` Protocol defined in
`helios_client.adapter`, so callers can swap implementations without code
changes.

## Context-window discipline

The codebase assumes that contributors (esp. AI agents) work under explicit
context-window budgets:

- **40–60% utilization target.** When the current window crosses 60%,
  compact intentionally (FIC — Frequent Intentional Compaction) before the
  auto-compact kicks in unpredictably.
- **Sub-agent isolation.** Long bridge runs (live live live live live live
  expeditions) should be invoked via the Agent / Workflow tools, not in the
  parent context. The Expedition orchestrator already respects this — it
  awaits per-baseline bridges sequentially in a single async loop.
- **Memory writeback.** Non-obvious learnings live under
  `~/.claude/projects/-home-admin-gh-agentdex-cli/memory/`; that directory
  is the canonical "what we learned from the last run" sink and is consulted
  by `/MEMORY.md` automatically.
- **STATE.md as the single source of project state.** `.supergoal/STATE.md`
  carries phase progress + Notable events; per-conversation tasks are
  ephemeral.

## Bridge contract — `BridgeResponse` dataclass (post PR #1 SF5)

`LongRunningCliBridge.send()` returns a frozen-slots `BridgeResponse`
dataclass exporting `text`, `langfuse_trace_id`, `cost_usd`, `tokens` as
first-class fields rather than relying on `bridge.last_cost_usd` /
`bridge.last_tokens` instance attrs that earlier callers had to
`getattr` out-of-band. The legacy properties remain as a read-only debug
mirror so ad-hoc REPL inspection still works, but every orchestrator +
stub + test must consume cost / tokens through the dataclass — see
DEFERRED.md SF5 closing entry. Cold-fallback path (PR #15) parses
claude's `--output-format json` JSON-array shape and propagates
`cost_usd` / `tokens` from the terminal `type=result` frame so the
fallback no longer leaves the orchestrator on the 4-char/token
heuristic.

## Judge call resilience — retry classifier (PR #18 + PR #20)

Every SDK call inside `oracle/soft.py:_invoke_judge` (anthropic /
openai / google-genai) is wrapped in `_call_judge_with_retries(fn,
label)`. The helper runs up to 3 attempts with 2-4-8 s exponential
backoff against the open-ended classifier
`_is_retryable_judge_error`:

  - retry on `InternalServerError` / `APIConnectionError` /
    `APITimeoutError` / `RateLimitError` (class-name match) and on any
    exception whose `repr` carries 5xx markers (520..527, 502, 503,
    504);
  - skip retries when the body carries explicit `"retryable":false` or
    `"owner_action_required":true` — Cloudflare 525 origin-SSL failures
    surface immediately so the orchestrator can move on instead of
    burning the per-baseline window;
  - re-raise the original exception class after the budget exhausts so
    the orchestrator's per-baseline `try/except` still excluded-fails
    the baseline cleanly.

Synchronous on purpose: judge runs inside `asyncio.to_thread` from the
orchestrator, so the blocking `time.sleep` between attempts does NOT
stall the event loop.

## Bridge spawn timeouts (PR #14 + PR #19)

Claude bridge handshake budget reads `cfg.spawn_timeout_sec` with a
30 s floor; `build_bridge("claude", ...)` sets `spawn_timeout_sec=60.0`
to cover SessionStart-hook latency in heavy project trees. Cold-shot
argv now carries `--dangerously-skip-permissions` (matching long-lived
mode) so a stdin permission prompt cannot silently hang the fallback;
`CliDead` errors surface `rc=<returncode>` + stdout + stderr excerpts
for diagnosability when one of those streams is empty.

## Async co-opetition (合作竞争), not real-time race

Per **ADR-0009 §Amendment-2026-06-08**: baselines run sequentially against
the same task; the Pareto judge ranks them after-the-fact. There is no
synchronous side-by-side battle. Bridges expose an async `send()` that the
orchestrator awaits one baseline at a time.

## Membership gate call order (post Phase 11b PRs #101–#105)

Per **ADR-0011** the arena ships a per-owner monthly membership primitive
that gates only enrichment endpoints (badge SVG, signed replay, bulk API
export, regression gate — 11c..11f). The free `/ladder` surface and every
existing route (battle / replay / evolution / fork / dispute / whoami /
methodology / skill.md / enrollment) **stay free forever** — anti-pay-to-rank
is encoded as the absence of `@require_membership` on those routes.

Call order on every gated endpoint:

```python
claims = authority.verify(token, scope=X)        # 1. signature + expiry + scope
authority.verify_membership(claims)              # 2. paid-feature gate (11b+)
authority.spend_quota(claims, scope=X)           # 3. daily cap
# ... then run the business logic
```

Steps 1 + 3 are pre-existing; step 2 is new in 11b and called ONLY by
paid-feature route handlers. Free-tier handlers MUST NOT call
`verify_membership`. The integration test
`test_skill_md_does_not_mention_admin_surface` asserts that operator
surfaces stay invisible to agent clients.

**Membership keyed by normalized owner**, not by `token_id`. This is
load-bearing for two reasons:

1. **Survives 7-day token rotation** — users re-enrolling do not lose
   membership; the same owner email gets the same membership row.
2. **Single code path for revocation** — re-grant with `valid_until_epoch
   <= now` is the revocation primitive. The EventLog records both grant
   events truthfully (audit preserved).

**Admin surface is operator-only.** `POST /admin/grant-membership` accepts
`X-Admin-Token` (SHA-256-hashed env var); plaintext NEVER lives on the
server. See `docs/runbooks/membership-admin.md` for generate / set / grant /
revoke / rotate procedures. The admin endpoint is **absent from SKILL.md
by design** — agent clients are untrusted-by-default and must not learn of
privileged surfaces. This is enforced by an integration test.

## Badge signing key + mint call order (Phase 11c, PRs #129/#130/#132)

Per **ADR-0011 §11c** the first paid feature is a verified-badge SVG
embeddable in third-party READMEs. The mint endpoint is the second gated
surface (after `/admin/grant-membership`); the SVG render + verify
endpoints are PUBLIC by design — the badge_token signature IS the auth.

`BadgeAuthority` holds a SEPARATE Ed25519 keypair from `ConsentAuthority`
(env: `ARENA_BADGE_SIGNING_KEY_HEX`; the consent env is unchanged at
`ARENA_SIGNING_KEY_HEX`). A leaked badge key lets an attacker mint fake
badges but consent tokens remain trustworthy; the inverse holds too. This
is D2 blast-radius isolation, ratified by user 2026-06-15. See
`docs/runbooks/badge-admin.md` for generation / rotation / emergency
revocation.

Call order on `POST /badge/mint` (the only gated badge endpoint):

```python
if gateway.badge_auth is None:                   # 0. degraded boot → 503
    raise _opaque_error(503, "badge mint not configured")
claims = authority.verify(token, scope="badge_mint")   # 1. sig + expiry + scope
authority.verify_membership(claims)              # 2. paid-feature gate
authority.spend_quota(claims, scope="badge_mint")     # 3. per-agent daily cap
badge_token = badge_auth.sign_badge({...})       # 4. mint (Ed25519)
```

The 503 in step 0 is distinct from the 403 / 404 paths so operators can
tell key-missing-at-boot from owner-fault at runtime.

`spend_quota(scope="badge_mint")` keys on `claims.agent_name` per §3b §5e
(survives `/enroll/reissue`, segregates across an owner's multiple
agents). The only scope that keys on owner is `battle` — the §3a
rotation-as-reset closure.

Public render endpoints (`GET /badge/<agent>/<token>.svg` and
`/badge/<agent>/<token>/verify`) carry NO consent token, NO membership
lookup. Both call `BadgeAuthority.verify_badge` then `gateway.ladder_public()`
— the same source `/ladder` uses. Any membership-derived rating boost
would diverge from `/ladder` and fail the Q5 property test. Failure modes
collapse to opaque `404` (D7 anti-enumeration); only the degraded
`badge_auth is None` path returns `503`.

**Badge admin surface stays operator-only**, same posture as the
membership admin endpoint. `ARENA_BADGE_SIGNING_KEY_HEX`, the rotation
procedure, and emergency revocation MUST NOT appear in `SKILL.md`,
`ENROLLMENT.md`, `METHODOLOGY.md`, or any agent-facing documentation.
The public `POST /badge/mint` endpoint IS documented to agents (an owner
surface); the key-custody surface is not.

## Glossary

- **Three Cards** — TaskCard (frozen input), ResultCard (per-baseline
  output), EvolutionCard (across-baseline mutation seeds). All pydantic
  strict + `extra="forbid"`.
- **Oracle layer** — `hard` (regex + tolerance), `soft` (Langfuse-wrapped
  Anthropic SDK as judge), `repair` (meta-oracle emitting structural
  seeds).
- **Pareto verdict** — winner / no_clear_winner across
  (pass_rate ↑, cost ↓, speed ↓).
- **Mutation seed** — a pydantic Seed carrying `kind`, `description`,
  `confidence`, `seed_provenance ∈ {"structural", "learned"}` (R6
  truth-in-advertising).

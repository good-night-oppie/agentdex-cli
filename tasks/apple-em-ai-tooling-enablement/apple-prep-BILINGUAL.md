## 11. One-page glance / 一页速查

### Five pillars (BENE2-DESIGN.md §2) / 五大支柱

1. **Parity core** — eval discipline rebuilt on the engram substrate: pre-registered falsifiable probes (ACCEPT/REJECT/VOID), experiments journal, consolidation ("dream") promotion passes, outcome-recency plasticity.
2. **Evolution engine** — structured text genomes + reflective mutation + Pareto frontier + trace→skill distillation; kill-gated promotion (nothing promotes without an ACCEPT verdict; verifier process-isolated).
3. **Memory & context OS** — granules = ladder tiers 0–3, familiarity-gated fast/slow retrieval, token-budget context manifests, pollution detection → clean-context recovery.
4. **Harness-engineering layer** — enforced autonomy ladder, agent-senses manifest, debt sweeper, loop guards, single-intent tool discipline.
5. **Trust & experience** — trust ledger (4 deterministic signals), provenance required on every engram, spec-as-artifact, <60s keyless `bene demo`; **the DevEx hook**.

1. **对等核心** — 在 engram 基底上重建评测纪律：预注册的可证伪探针（ACCEPT/REJECT/VOID）、实验日志、整合（"做梦"）晋升流程、按结果-近因加权的可塑性。
2. **演化引擎** — 结构化文本基因组 + 反思式变异 + Pareto 前沿 + trace→技能蒸馏；kill-gated 晋升（没有 ACCEPT 裁定就不得晋升；验证器与演化器进程隔离）。
3. **记忆与上下文操作系统** — 颗粒即阶梯 0–3 层、熟悉度门控的快/慢检索、token 预算的上下文清单、污染检测 → 干净上下文恢复。
4. **Harness 工程层** — 强制执行的自治阶梯、agent senses 清单、债务清扫器、循环防护、单一意图工具纪律。
5. **信任与体验** — trust ledger（4 个确定性信号）、每条 engram 强制 provenance、规格即工件、<60 秒免密钥 `bene demo`；**DevEx 钩子**。

### Three lenses (one signature idea each) / 三大师视角

- **Hassabis** — falsifiable gates / games-as-testbeds: every mechanism that evolves is gated by a pre-registered, falsifiable probe.
- **Sutskever** — compression-is-understanding: the engram ladder (raw trace → episodic → semantic → procedural → strategic) is literal compression of experience.
- **Karpathy** — march-of-nines: the L0–L4 autonomy ladder — "each step up is a nine"; grants are verified artifacts, per-agent and revocable.

- **Hassabis** — 可证伪门槛 / 游戏即试验场：每个会演化的机制都由预注册的可证伪探针把关。
- **Sutskever** — 压缩即理解：engram 阶梯（原始轨迹 → 情景 → 语义 → 程序 → 战略）就是对经验的字面压缩。
- **Karpathy** — 九的征程：L0–L4 自治阶梯——"每上一级就是一个九"；授权是经验证的工件，按 agent 颗粒、可撤销。

1. Qumulo 时期的 the 0.1.0 predecessor harness — 测试分诊 / on-call 工作负载的 agent 编排
2. 离开 Qumulo 后的两个框架 — KAOS v0.9.1 与 BENE v0.1.0

4. 挖掘文献 — 约 100 篇研究摘要 → 48 篇引用（`docs/research/SYNTHESIS.md`）

6. 信任支柱 — 工程师只在信任的前提下采用 agent 工具（第五支柱，DevEx 钩子）

### Top-5 defense-card questions / 五大防守问题

1. What actually works today vs. what is only designed? → honesty rule below.
2. Why one substrate — "everything is an engram"? → fixes eval-layer-can't-write-to-memory split (GAP-AUDIT KAOS-6), provenance joins everything.
3. How do you stop self-evolution from reward-hacking? → kill-gated promotion; verifier process-isolated from the evolver (AEVO evidence in BENE2-DESIGN.md §2).
4. When does an agent get more autonomy? → L0–L4: falsifiable gates + trust-ledger thresholds, per-domain, revocable (§3).
5. Why does this matter to Apple DevEx? → trust ledger + provenance everywhere + probe-tested experience claims = the adoption problem, solved as architecture.

1. 今天真正能跑的是什么、哪些只是设计？→ 见下方诚实规则。
2. 为什么是单一基底——"一切皆 engram"？→ 修复评测层无法写回记忆层的割裂（GAP-AUDIT KAOS-6），provenance 把一切连起来。
3. 如何防止自我演化沦为 reward hacking？→ kill-gated 晋升；验证器与演化器进程隔离（BENE2-DESIGN.md §2 的 AEVO 证据）。
4. agent 什么时候获得更高自治权？→ L0–L4：可证伪门槛 + trust ledger 阈值，按领域、可撤销（§3）。
5. 这对 Apple DevEx 为什么重要？→ trust ledger + 全链路 provenance + 经探针验证的体验承诺 = 把"采用难题"用架构解决。

### Honesty rule / 诚实规则

**BENE 0.1.0 works today** — 445 passing tests, 37 MCP tools, VFS-per-agent SQLite, checkpoints/diff/restore, FTS5 memory+skills, LogAct shared log, tier router with 5 providers, metaharness search with Pareto frontier, Temporal runtime, storage protocol (SQLite+Postgres), web UI + TUI, obsidian export. **BENE 2.0 is a design** — designed 2026-06-11, build in flight, phases 4–9 pending. Never present a 2.0 feature as working.

**BENE 0.1.0 今天就能跑** — 445 个通过的测试、37 个 MCP 工具、每 agent 独立 SQLite VFS、checkpoint/diff/restore、FTS5 记忆+技能、LogAct 共享日志、5 个 provider 的分层路由、带 Pareto 前沿的 metaharness 搜索、Temporal 持久运行时、存储协议（SQLite+Postgres）、web UI + TUI、obsidian 导出。**BENE 2.0 是设计稿** — 2026-06-11 完成设计，构建进行中，第 4–9 阶段待完成。绝不把 2.0 特性说成已可用。
